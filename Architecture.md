# Architecture

## Overview

This project uses a **Facade + Domain Service Layered Architecture**.  
It shares ideas with Hexagonal and DDD but does not enforce Port/Adapter interfaces.  
The goal is pragmatic separation: domain logic stays pure, facade handles only flow.

---

## Layer Definitions

### Domain Service
- Owns the business logic of a single domain
- Accesses the database only through its own domain's Repository
- Must not depend on another domain's Repository or Service
- Does **not** need to own a Repository to qualify as a domain service — pure logic with no I/O is still a domain service

### Facade Service
- Composes multiple domain services to implement a feature
- Responsible for **flow only**: call A, then call B, return result
- Must not contain decision logic, transformation logic, or data mapping
- Must not directly reference any Repository class

### Controller
- Calls Facade Services only
- Must not call Domain Services directly
- May contain no logic beyond request/response mapping

---

## Rules

| Rule | Details |
|------|---------|
| Controller → Facade only | Controllers must not call domain services directly |
| Facade → Domain Services only | Facade must not reference Repository classes |
| Domain → own domain only | Domain services must not call another domain's Repository or Service |
| Repository = not required for domain service | Pure logic (no I/O) still belongs in a domain service if it represents a domain concept |

---

## Intended Direction

### Facade as flow-only orchestrator
The facade should read like a sequence of steps:
```
1. find hotel place
2. resolve flight info
3. create trip
4. generate date plans
5. generate place plans
6. generate transport plans
```
No branching logic, no data transformation, no algorithm — just delegation.

### Domain services as isolated units
Each domain service should be testable in isolation.  
Cross-domain knowledge does not belong in a domain service.  
A method that decides *what* to do (e.g. which plans are mandatory, how to distribute places across dates) is domain logic — regardless of whether it touches a Repository.

## Violations Found

### 1. Controller calling Domain Services directly

**`TripController.createTrip()`**  
Directly calls `AutoCreatePlanService` and `PlaceFacadeService` instead of routing through `TripFacadeService`.  
The entire async plan creation chain belongs inside `TripFacadeService.createTrip()`.

```
// Current (wrong)
Controller → PlaceFacadeService.findByGoogleId()
Controller → AutoCreatePlanService.autoGenerateDatePlans()
Controller → AutoCreatePlanService.autoGenerateAllPlans()
Controller → AutoCreatePlanService.autoGenerateAllThemeTransportPlans()

// Should be
Controller → TripFacadeService.createTrip()
               └→ PlaceFacadeService / AutoCreatePlanService (internally)
```

**`PlaceController.resolveWeather()`**  
Controller holds weather code interpretation logic and directly calls `OpenMeteoService`.  
This is both a rule violation and misplaced domain logic.

```
// Should be
Controller → PlaceFacadeService.recommendPlace()
               └→ WeatherResolveService.resolve() (new domain service)
```

---

### 2. Domain logic living inside Facade

**`PlaceFacadeService.recommend()` + `selectTopFiveWithAi()`**  
These private methods (77 lines total) implement a complete recommendation algorithm:
- Filter candidates from DB
- Compute routes via Google API for each candidate
- Call AI to select top 5
- Handle fallback

This is domain logic, not flow. It should be extracted into a `PlaceRecommendService` domain service.  
The facade method becomes: find context → delegate to `PlaceRecommendService` → return result.

**`TripFacadeService.resolveFlightInfo()`**  
Parses flight datetime strings and looks up the airport Place.  
This is data resolution logic, not flow orchestration.  
Should move to a dedicated service or to the DTO itself.

**`PlanFacadeService.getPlans()`**  
Groups DatePlans by theme, flattens and sorts PlacePlans — this is a query transformation.  
Should be delegated to `PlanSelectService` as a dedicated method.

---

### 3. Domain logic hidden as private methods in AutoCreatePlanService

`AutoCreatePlanService` is heavy because logic that should be domain services was kept inline as private methods under the assumption that *"a domain service must own a Repository."*

| Private method | What it really is | Where it should go |
|---|---|---|
| `buildNecessaryPlans()` | Decides mandatory plans (airport/hotel) by date rules | `DatePlanBuildService` |
| `buildDatePlanDtos()` | Assembles DatePlan DTOs from dates/regions/places | `DatePlanBuildService` |
| `buildCandidates()` | Fetches place candidates by type for a date | `PlanCandidateService` |
| `getNecessaryPlaces()` | Parses `googleIds` string → fetches places | `PlanCandidateService` |
| `toPlacePlanInfos()` | Maps PlacePlan entities to DTOs | `PlanCandidateService` |
| `getPlacedPlaceIds()` | Collects placed place IDs for hotel exclusion | `PlanCandidateService` |
| `generateTransportPlans()` | Computes routes + creates TransportPlan entities | `TransportPlanGenerateService` |

If these are extracted, `AutoCreatePlanService` becomes a true facade — each public method reads as a pure sequence of delegations.

---

### 4. Domain Service using Repository from another domain

**`WeatherService`** directly injects `RegionRepository` instead of using `RegionQueryService`.  
Region data access belongs to the Region domain; WeatherService should delegate.
